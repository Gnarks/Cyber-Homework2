\code{"Nope, I do not accept this..."} as output.

By putting the right argument, the program loops 60 times and prints \code{The world is against me, please adapt it...}\\ every 0.5 seconds (nanosleep of 0x1dcd6500 == 500,000,000 ns) before printing \code{I need a rest}.


This led me to reason about the permissions of the /tmp folder, but that wasn't the issue.
In fact, it was simply that the \code{Heartsnatcher} directory in \code{/tmp} didn't exist and was causing the issue.
As, the binary couldn't write the \code{novel} file in a non-existing directory.
Once the \code{/tmp/Heartsnatcher} directory created, the output of the binary was \code{20} indicating it wrote 20 bytes (the flag).

By looking at the assembly, we can figure out that there is a gap of 0.5s between the creation of the file and a \code{chmod} call to change its permissions.
This gap could be exploited by using a Race-condition attack.
To this end, the \code{Boris\_RaceCondition.c} code is used to read in a loop the \code{/tmp/Heartsnatcher/novel} file before it changes permissions. 

Running this script in a separate terminal then executing the binary with the \code{/tmp/Heartsnatcher/novel} argument gives the result illustrated in Figure \ref{fig:borisFlag}.

This provides the flag : \code{sinfo044\{Armstrong\}} which is explained in Section \ref{BorisLore}.

\section{John Nash}\label{John}
At first glance, the structs and free functions give a glimpse of a possible free after use attack to exploit the binary. 
Looking at it in details, we find that both structs are of the same size, as in their respective creation functions, 2 bytes are allocated for them.
This and the fact that the free function doesn't clear the struct data but only frees the pointer indicates that by creating an addition, freeing it and creating a division might result in some values from the addition to be passed to the division (those which aren't overwritten by the division). 

To better grasp the allocation of both structs, Table \ref{tab:john} compares them byte by byte.

This is valuable insights but how to break the equilibrium stays a mystery.
Using gdb to look at the \code{break\_equilibrium} function reveals that the res should be negative with  \code{34}(in decimal) as status.   
This is true for both structs meaning the addsub (resp. div) struct the represented signed integer at 0x10 offset (resp. 0x18) should be negative with the 0x18 offset (resp. 0x17) equal to 34. 

Now that we have a better understanding of the allocation and the win condition, we'll look at how we could arrive in a winning status. 
Exploring the fact that the status of an addsub corresponds to the first byte of the \code{res} of a div, we can leverage it to put a \code{status} 34 in an addsub.
Moreover, the last byte of the addsub \code{res} corresponds to the div \code{status}. 

By looking at the \code{create} functions of any struct, we observe that it only overwrites the  $0x0 \rightarrow 0x10$ bytes.
In addition, the \code{evaluation} also have a simmilar logic for both structs : if the result is negative, the \code{res} isn't overwritten and \code{-70} is set as status. 
If the result is positive, the status is set to 34 and the \code{res} is overwritten. 
The div \code{options} and addsub \code{is\_sub} aren't relevant for our exploit.

Finaly, knowing what the allocation is and the logic of the different function for each struct we can leverage this information to attack the binary using a free-ater-use attack. 
The free-after-use is usable as, due to their similar size, the allocator will simply allocate the same space after a free for div and addsub structs.   
This means that, by setting values for a struct (creating and evaluating it), freeing it and recreating a new struct on top of it, we are able to keep the values not overwritten from the previous struct. 

Conviently, the status for negative number is a negative number itself, and is the most significant byte of the addsub \code{res}.
Moreover, a div \code{res} of 34 would result in an addsub \code{status} of 34. 

Using those two observations, we create a serie of instruction to break the equilibrium : 
\begin{enumerate}
	\item create a division that would result in 34 and evaluate it 
	\item free it
	\item create another division that would result in a negative number and evaluate it
	\item free it
	\item create an addition with arbitrary values 
	\item break the equilibrium
\end{enumerate}

(1) sets the addsub \code{status} to 34.
(3) sets the addsub \code{res} to a negative value. 
(5) switches the div struct to an addsub one.
(6) results in an equilibrium break as the win condition is satisfied.

This sequence breaks the equilibrium and reveals the flag of the binary (\code{sinfo044\{MonroeTownship\}}) as seen in Figure \ref{fig:flagjohn}.

The \code{john\_free\_exploit.sh} script can be used to directly reproduce the sequence and retrieve the flag.

\section{Arthur Scherbius}


The binary here, welcomes us and expects 2 inputs, a username and a password.
The welcome message is \code{Welcome to Enigma v20.2.6 SuperDoudou} \code{Edition!}
followed by a message asking us to enter the username or password. By using,
\code{strings} to see the readable strings of the file, we can see that there
is indeed the input asking strings, asking us to enter a value, but no signs of the
welcome message. Using gdb, we can find some instruction decrypting a value,
\code{<rapterrg>}, before printing it and then print the input asking strings.
We can conclude that rapterrg correspond to an encrypted version of the welcome
message, by finding the value of rapterrg we know have a plain text and its
cypher text. By using gdb, we can also see that the code use AES-GCM as its
encryption algorithm, and so use a XOR between a key and a plaintext to get the
cypher one. This logic can be used in another way, by XORing the cypher text
and the plaintext we can get back the key used for the encryption.

By doing this technic, we can get the key used for the encryption of the
Welcome message, and if this key is used anywhere to encrypt we can get the
plaintext easily by XORing the key with it. But using gdb again we can see that
after passing any input asked, it is encrypted and compared to a value,
\code{<raphfre>} for the user and \code{<rapcnff>} for the password.

By retrieving these value, we can XOR them with the key found earlier and if
the key has been reused we should get the plaintext of the wanted input. 

Here the key is reused both for the username and the password, and so we get: 


\begin{itemize}
  \item the username : \code{WhatAW0nderfulTeacher}
  \item the password : \code{Th1sIsAV3ryComPleXPasswordToGuess}
\end{itemize}

By putting these informations as inputs we get the flag : \code{sinfo044\{AlanTuring\}}

\subsection{Fairy Tail Buffer Overflow}

Using objdump and gdb, we can detect functions in the code, \code{main},
\hbox{1em}
\code{onePieceIsForTheWeak} and \code{dragonBallForever}. Here only,
\code{main} and \code{onePieceIsForTheWeak} are callable, but no flags are to be
found here. Analyzing \code{dragonBallForever}, we can see that it
creates a value in the stack, to be able to see it, we need to call this function.

As the stack is executable, a buffer overflow is possible. To do so, we need  to find the
address of the function we want to call, the difficulty here was to find the good address 
of \code{dragonBallForever}. Fortunately, the message printed by the binary gives us an
address which is the \code{onePieceIsForTheWeak} one. 

As the difference between their address is constant we can easily find 
the address of the wanted function. But if we practice the overflow with 
"random"  chosen characters, we get this message \code{Good try one-punch man},
so we need to find another way to do so. By analyzing \code{dragonBallForever}, we see
 that two values are checked, \code{megaman} and \code{deathnote}, which are modified in \code{onePieceIsForTheWeak} if \code{FairyTail} is written as input. By filling the first value of our buffer overflow string with \code{FairyTail} instead of random values, we finally get the first wanted flag.
The flag given is \code{sinfo044\{TheRealAnswerIsSailorMoon\}}.
